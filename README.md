## 📅 9주차: UI 트리 구조의 이해, JSX 스타일링 및 이벤트 상호작용 (4월 29일)

### 1. UI를 트리 구조로 이해하기

#### (1) Render 트리 (Render Tree)
- **개요**: React 컴포넌트 간의 부모-자식 관계를 나타내는 계층 구조입니다.
- **특징**: 
  - 앱의 컴포넌트들이 어떻게 중첩되어 화면을 구성하는지 보여줍니다.
  - 상위 컴포넌트가 하위 컴포넌트를 호출하는 '렌더링 관계'를 시각화한 지도로, 성능 최적화와 상태 전달 범위를 파악하는 기준이 됩니다.

#### (2) 모듈 의존성 트리 (Module Dependency Tree)
- **개요**: 프로젝트 내 파일(`import`/`export`) 간의 호출 관계를 나타내는 트리입니다.
- **차이점**: 
  - Render 트리는 컴포넌트의 렌더링 순서만 다루지만, 모듈 트리는 컴포넌트가 아닌 일반 JS 파일(상수 데이터, 유틸리티 함수 모듈 등)까지 모두 포함합니다.
  - 프로젝트의 전체적인 파일 설계 구조를 파악하는 데 사용됩니다.

---

### 2. 이벤트와 상호작용 (Interaction)

#### (1) 개요
- 화면을 구성하는 요소 중에는 사용자의 입력에 반응해서 업데이트되는 요소가 있습니다.
- 예를 들어, 이미지 갤러리에서 특정 이미지를 클릭하면 해당 이미지가 활성 상태가 되는 것과 같습니다.

#### (2) 이벤트에 응답하기
- React에서는 JSX 요소에 직접 **이벤트 핸들러**를 추가할 수 있습니다.
- **이벤트 (Event)**: 사용자의 클릭, 타이핑, 마우스 오버와 같은 특정 입력을 의미합니다.
- **이벤트 핸들러 (Event Handler)**: 이벤트가 발생했을 때 반응하여 실행되는 로직(함수)입니다.



#### (3) 주요 이벤트 종류
| 종류 | 속성명 (React) | 설명 |
| :--- | :--- | :--- |
| **클릭** | `onClick` | 요소가 클릭되었을 때 발생 |
| **마우스 호버** | `onMouseEnter` / `onMouseLeave` | 마우스 커서가 요소 위로 올라오거나 벗어날 때 발생 |
| **폼 입력** | `onChange` | input, textarea 등의 값이 변경될 때 발생 |
| **포커스** | `onFocus` / `onBlur` | 요소가 포커스를 얻거나 잃었을 때 발생 |

---

### 3. 빌드와 번들러 (Bundler)

- **번들러의 역할**: 모듈 의존성 트리를 따라가며 앱 실행에 필요한 모든 파일을 추적하고, 이를 하나로 묶어 브라우저가 빠르게 읽을 수 있도록 최적화(압축 및 빌드)하는 도구입니다.
- **중요성**: 프로젝트 규모가 커짐에 따라 늘어나는 파일 다운로드 비용을 줄이고, UI 지연 시간을 최소화하여 사용자 경험을 개선하는 필수 단계입니다.

---

### 4. JSX 스타일링의 원리와 가독성

#### (1) 인라인 스타일링 (Inline Styles)
- **원리**: JSX 요소에 직접 JavaScript 객체 형태로 스타일 속성을 부여하는 방식입니다.
- **핵심 규칙**: 속성 이름은 반드시 **카멜 케이스(camelCase)**로 작성해야 하며, 데이터나 상태 변화에 따라 스타일을 실시간으로 변경해야 할 때 유용합니다.

#### (2) 외부 CSS와 가독성 관리
- **외부 파일 활용**: 고정된 디자인은 별도의 CSS 파일로 관리하며, JSX에서는 `className` 속성을 통해 스타일을 적용합니다.
- **컴포넌트 분리**: 코드의 길이가 길어지면 기능을 별도 모듈(파일)로 분리하여 관리해야 합니다.

---

### 5. CSS Module: 고유한 로컬 스코프

#### (1) 일반 CSS의 한계
- **전역 오염**: 일반 CSS는 전역으로 선언되므로, 다른 컴포넌트에서 같은 클래스명을 사용할 경우 스타일이 충돌할 위험이 큽니다.

#### (2) CSS Module의 해결 방식
- **자동 변환**: 빌드 시 클래스 이름을 `[파일이름]_[클래스이름]__[해쉬값]` 형태로 자동 변환하여 로컬 스코프를 제공합니다.
- **사용 방법**: 
  - 클래스명이 하이픈을 포함한 **kebab-case**일 경우 `styles['main-container']`와 같이 **대괄호 표기법**으로 접근해야 합니다.

### ✅ 9주차 최종 핵심 요약
- **구조**: 렌더링 계층(Render 트리)과 파일 연결(모듈 트리)을 구분합시다.
- **상호작용**: 클릭, 호버, 입력 등 다양한 사용자의 **이벤트**에 대응하는 **이벤트 핸들러**를 작성하여 동적인 앱을 만듭니다.
- **스타일링**: CSS Module을 활용해 클래스 충돌을 방지하고, 명명 규칙에 맞는 접근 방식을 사용합시다.

## 📅 7주차:  (4월 15일)

### 1. 배열 데이터의 구조화 (Structuring Data)
- **개요**: 단순히 문자열로 이루어진 배열보다 객체(`Object`)를 포함한 배열로 데이터를 관리하면 더욱 강력하고 유연한 UI 구조화가 가능합니다.


### 2. 배열의 항목 필터링하기: filter() 함수

- **개요**: JavaScript의 `filter()` 함수를 사용하여 전체 데이터 중 특정 조건에 맞는 데이터만 골라내어 **새로운 배열을 반환**할 수 있습니다.
- **동작 원리**:
  1. 배열의 각 항목을 순차적으로 받아 지정된 조건을 테스트합니다.
  2. 테스트 결과가 **`true`**인 항목들만 모아서 새로운 배열을 생성합니다.
  3. 테스트 결과가 **`false`**인 항목은 결과 배열에서 제외됩니다.

### 3. 화살표 함수(Arrow Function)의 반환 규칙

React에서 리스트를 렌더링할 때 사용하는 화살표 함수는 작성 방식에 따라 `return`문의 필요 여부가 달라집니다.

#### (1) 묵시적 반환 (Implicit Return)
- 화살표(`=>`) 바로 뒤에 식을 작성하면 해당 값이 **자동으로 반환**됩니다.
- 별도의 중괄호 `{ }`나 `return` 키워드가 필요하지 않아 코드가 간결해집니다.

```jsx
const listItems = chemists.map(person =>
  <li>...</li> // 묵시적 반환 (자동으로 return 됨)
);
```
-
#### (2) 명시적 반환 (Explicit Return) - "Block Body"
- 화살표(`=>`) 뒤에 **중괄호 `{ }`**가 오는 경우를 **"Block body"**를 가졌다고 말합니다.
- 이 경우에는 반드시 **`return`문을 명시적으로 작성**해야 합니다.
- `return`문을 작성하지 않으면 아무것도 반환되지 않아 화면에 출력되지 않습니다.

```jsx
const listItems = chemists.map(person => { // 중괄호 사용 시
  return <li>...</li>; // 명시적 반환 (반드시 return 작성)
});

```
- 특징: 중괄호를 사용하는 방식은 함수 내부에서 return 전까지 한 줄 이상의 로직(변수 선언, 계산 등)을 작성할 수 있다는 장점이 있습니다.

### 4. 컴포넌트의 순수성 유지 (Keeping Components Pure)

React는 모든 컴포넌트가 **'순수 함수(Pure Function)'**라고 가정하고 설계되었습니다. 이는 컴포넌트가 안정적으로 예측 가능하게 동작하도록 만드는 핵심 원칙입니다.

#### (1) 순수 함수의 특징
- **동일 입력, 동일 출력**: 같은 입력값(Props)이 주어지면 항상 동일한 JSX를 반환해야 합니다.
- **사이드 이펙트(Side Effect) 없음**: 함수 실행 전이나 외부의 객체, 변수를 수정하지 않습니다. (함수 호출만으로 외부 상태가 변해서는 안 됨)

#### (2) 렌더링을 순수하게 유지해야 하는 이유
- **예측 가능성**: 동일한 Props에 대해 항상 같은 화면을 보장하므로 버그 추적이 쉽습니다.
- **최적화**: 데이터가 변경되지 않았다면 React가 렌더링을 건너뛸 수 있어 성능이 향상됩니다.
- **안전성**: 컴포넌트 렌더링 도중 외부 데이터를 직접 수정하면, 다른 컴포넌트의 렌더링 결과가 오염될 수 있습니다.

#### (3) 주의사항: 렌더링 단계에서의 데이터 변형 금지
- 렌더링은 "계산"하는 과정이어야 합니다. 렌더링 도중에 외부 변수를 수정하거나 배열에 `push`를 하는 등의 행위는 순수성을 깨뜨리는 행위입니다.
- **해결책**: 외부 상태를 변경해야 한다면 렌더링 과정이 아닌 **이벤트 핸들러**나 **useEffect** 내부에서 수행해야 합니다.

### 5. 사이드 이펙트(Side Effects)의 이해와 처리

React 컴포넌트는 기본적으로 순수해야 하지만, 실제 앱에서는 서버에서 데이터를 가져오거나 브라우저 API를 사용하는 등 외부 시스템과의 연결이 반드시 필요합니다. 이러한 변화를 **'사이드 이펙트(Side Effect)'**라고 합니다.

#### (1) 사이드 이펙트란?
- **정의**: 렌더링 과정 이외에 발생하는 프로그램의 상태 변화를 의미합니다.
- **예시**: 
  - 네트워크 요청 (API 호출)
  - DOM 요소 직접 조작 (스크롤 위치 변경, 포커스 등)
  - 타이머 설정 (`setTimeout`, `setInterval`)

#### (2) 사이드 이펙트가 발생하는 위치
컴포넌트의 순수성을 깨뜨리지 않기 위해 사이드 이펙트는 반드시 다음 두 곳 중 하나에서 발생해야 합니다.

1. **이벤트 핸들러 (Event Handlers)**:
   - 사용자의 클릭, 입력 등에 반응하여 실행되는 함수입니다.
   - 렌더링 도중 실행되는 것이 아니므로 순수할 필요가 없으며, 상태를 변경하기에 가장 적합한 장소입니다.

2. **useEffect Hook**:
   - 렌더링이 모두 완료된 후에 실행되는 특수 함수입니다.
   - 적절한 이벤트 핸들러를 찾을 수 없을 때(예: 컴포넌트가 나타나자마자 데이터를 불러와야 할 때) 최후의 수단으로 사용합니다.

#### (3) 왜 렌더링 중에 사이드 이펙트를 일으키면 안 되나요?
- **무한 루프 위험**: 렌더링 중에 상태를 변경하면 다시 렌더링이 발생하고, 이것이 반복되어 브라우저가 멈출 수 있습니다.
- **예측 불가능한 UI**: 렌더링 순서에 따라 화면이 이상하게 나올 수 있습니다.
- **성능 저하**: 불필요한 계산과 외부 통신이 반복되어 앱이 느려집니다.

---
### 6. 사이드 이펙트(Side Effect): 의도하지 않은 결과 방지하기

React에서 컴포넌트의 주 목적은 **Props와 State를 받아 JSX(화면)를 반환하는 것**입니다. 이 목적을 벗어난 모든 작업은 '부수 효과(Side Effect)'라고 부릅니다.

#### (1) 왜 '의도하지 않은 결과'인가?
- 컴포넌트는 오직 자신의 화면만 책임져야 합니다. 
- 만약 렌더링 도중 외부 변수를 수정하거나 API를 호출하면, 리액트가 화면을 다시 그릴 때마다 그 작업이 반복되어 데이터가 꼬이거나 앱이 멈추는 등 **예측할 수 없는(의도하지 않은) 결과**가 나타납니다.

#### (2) 대표적인 사이드 이펙트의 예시
- **데이터 요청**: 서버에서 데이터를 받아오는 행위 (fetch, axios)
- **수동 DOM 조작**: `document.querySelector` 등을 사용해 직접 HTML을 건드리는 것
- **타이머**: `setTimeout`이나 `setInterval` 사용
- **로그 기록**: 외부 서비스에 로그를 전송하는 것

#### (3) 사이드 이펙트를 안전하게 관리하는 법
리액트는 사이드 이펙트를 무조건 금지하는 것이 아니라, **"적절한 곳에서 실행하라"**고 권고합니다.

1. **이벤트 핸들러**: 사용자가 버튼을 클릭했을 때만 실행되므로 안전합니다.
2. **useEffect Hook**: 렌더링이 완전히 끝난 후 실행되므로 렌더링 과정을 방해하지 않습니다.

---

### 7. 지역 변경 (Local Mutation): 안전한 변형

컴포넌트의 순수성을 지키면서도 데이터를 가공해야 할 때 사용하는 기법입니다.

#### (1) 무엇이 문제인가? (Mutation vs Pure)
- **Mutation(변이)**: 렌더링 도중에 컴포넌트 **외부**에 있는 기존 변수나 객체를 수정하는 행위입니다. 이는 프로그램에 '돌연변이'를 일으켜 의도하지 않은 결과를 초래합니다.
- **Pure(순수)**: 순수 함수는 함수 스코프 외부의 어떤 것도 건드리지 않아야 합니다.

#### (2) 지역 변경(Local Mutation)이란?
- **정의**: 렌더링 과정 중에 **함수 내부에서 방금 생성된** 변수나 객체를 변경하는 것을 의미합니다.
- **안전한 이유**: 이 변경사항은 해당 컴포넌트 내부(지역적)에서만 발생하며, 외부의 다른 컴포넌트나 전역 상태에 전혀 영향을 주지 않기 때문에 '순수성'을 해치지 않습니다.

### 8. UI를 트리 구조로 이해하기 – Render 트리

React를 포함한 현대적인 UI 라이브러리들은 화면(UI)을 **트리(Tree) 형태**로 모델링하여 관리합니다.

#### (1) 왜 트리 구조인가?
- **관계 파악**: 컴포넌트 간의 부모-자식 관계를 시각적으로 이해하는 데 큰 도움이 됩니다.
- **개념 확장**: 향후 배우게 될 성능 최적화나 복잡한 상태 관리(State Management) 개념을 파악하는 기반이 됩니다.
- **보편성**: 브라우저(HTML/DOM, CSS/CSSOM)나 모바일 플랫폼에서도 이미 뷰의 계층 구조를 나타내는 데 트리를 사용하고 있습니다.

#### (2) Render 트리의 특징
- **요소 사이의 관계 모델**: UI는 단순히 나열된 조각이 아니라, 서로 얽혀 있는 관계의 모델입니다.
- **추상화**: 우리가 작성한 `Component A, B, C`는 React에 의해 트리 노드로 변환되고, 최종적으로 `ReactDOM`을 통해 브라우저가 이해할 수 있는 DOM 트리로 렌더링됩니다.

#### (3) 트리 구조의 이점
- **상태의 흐름**: 데이터(Props)가 위에서 아래로 어떻게 흐르는지 명확히 알 수 있습니다.
- **변경 범위 파악**: 특정 컴포넌트가 업데이트될 때, 트리의 어느 부분까지 영향을 받는지(재렌더링 범위)를 예측할 수 있게 해줍니다.

## 📅 6주차: 조건부 렌더링 및 리스트 렌더링 핵심 이론 (4월 8일)

### 1. 조건부 렌더링 (Conditional Rendering)
컴포넌트가 특정 조건에 따라 서로 다른 UI를 화면에 그려내는 기법입니다. React는 별도의 문법 대신 JavaScript의 논리 연산자와 조건문을 그대로 사용합니다.

#### (1) if 문을 통한 조건부 반환
- 가장 명확한 분기 방법입니다. 조건에 따라 전체 리턴문을 다르게 작성하여 완전히 다른 컴포넌트 구조를 보여줄 때 사용합니다.
- **특징**: 로직이 복잡할 때 컴포넌트 내부에서 가독성을 높여줍니다.

#### (2) 삼항 연산자 (`? :`)
- JSX 마크업 내부에서 직접 조건을 따질 때 유용합니다.
- **문법**: `{조건 ? 참일 때의 JSX : 거짓일 때의 JSX}`
- **장점**: 짧은 조건부 텍스트나 간단한 스타일 변경에 매우 효과적입니다.

#### (3) 논리 AND 연산자 (`&&`)
- "조건이 참일 때만 무언가를 보여주고, 그렇지 않으면 아무것도 표시하지 않음"을 구현할 때 사용합니다.
- **특징**: React는 `false`를 구멍(hole)으로 인식하여 화면에 렌더링하지 않습니다.

---

### 2. 변수에 조건부 JSX 할당 (가장 권장되는 방식)
- **개요**: `return`문 내부의 JSX가 조건문으로 인해 지저분해지는 것을 방지하기 위해 사용합니다.
- **원리**: JavaScript의 `let` 변수에 기본 JSX를 담아두고, `if` 조건문을 통해 필요한 시점에만 JSX 조각을 재할당하는 방식입니다.
- **장점**: 컴포넌트의 최종 구조(`return` 부분)를 한눈에 파악하기 쉬워지며 유지보수가 매우 용이합니다.

---

### 3. 리스트 렌더링 (List Rendering)
데이터 배열을 순회하며 반복되는 UI 요소(목록, 카드 등)를 효율적으로 생성하는 기법입니다.

#### (1) JavaScript map() 함수 활용
- 배열의 각 데이터를 순회하면서 새로운 JSX 엘리먼트 배열로 변환하는 핵심 도구입니다.
- **데이터 기반 설계**: 하드코딩 대신 데이터를 배열로 관리하고 `map()`으로 뿌려주는 방식이 현대 프론트엔드의 표준입니다.

#### (2) Key 속성의 역할과 필수성
- 리스트의 각 항목은 반드시 고유한 `key` prop을 가져야 합니다.
- **React의 동작 원리**: React는 가상 DOM을 비교할 때 `key`를 사용하여 어떤 항목이 변경, 추가, 삭제되었는지 식별합니다.
- **주의사항**: 
    - `key`는 형제 요소 사이에서만 고유하면 됩니다.
    - 배열의 `index`를 key로 쓰는 것은 리스트의 순서가 바뀔 때 버그를 유발할 수 있으므로, 데이터 고유의 `id`를 사용하는 것이 권장됩니다.

---



### ✅ 핵심 요약
- **조건부 UI**: 단순한 분기는 삼항 연산자나 `&&`를, 복잡한 로직은 변수 할당 방식을 사용합니다.
- **리스트 최적화**: `map()`을 통한 반복 렌더링 시에는 성능과 정확한 업데이트를 위해 반드시 고유한 `key`를 부여합니다.

## 📅 5주차: Props를 통한 컴포넌트 통신 (4월 1일)

### 1. 개요
- **정의**: Props(properties)는 React 컴포넌트들이 서로 통신하기 위해 사용하는 수단입니다.
- **역할**: 부모 컴포넌트가 자식 컴포넌트에게 데이터를 전달할 때 사용합니다.
- **유연성**: 객체, 배열, 함수를 포함한 모든 JavaScript 값을 전달할 수 있습니다.

---

### 2. 핵심 용어 정의
| 용어 | 정의 | 예시 |
| :--- | :--- | :--- |
| **태그 (Tag)** | 요소를 표시하기 위한 개별 기호 | `<div>`, `<li>` 등 |
| **엘리먼트 (Element)** | 여는 태그 + 내용 + 닫는 태그 전체 | `<p>내용</p>` |
| **어트리뷰트 (Attribute)** | HTML 태그 안의 정적인 속성 | `<img src="...">`의 `src` |
| **프로퍼티 (Property)** | DOM 객체 내의 동적인 속성 | JS로 제어되는 `input.value` |

- **차이점**: 사용자가 값을 입력하면 **Property**는 변하지만, HTML의 **Attribute**는 초기값 그대로 유지됩니다.

---

### 3. Props를 통한 컴포넌트 통신 원리
- **부모 컴포넌트**: 데이터를 전달하는 주체 (Parent).
- **자식 컴포넌트**: 받은 Props를 사용해 UI를 생성하는 객체 (Child).

#### ✨ Props의 3가지 특징
1. **단방향 흐름**: 데이터는 항상 부모에서 자식으로만 흐릅니다.
2. **읽기 전용**: 자식 컴포넌트는 받은 Props를 직접 수정할 수 없습니다.
3. **다양한 타입**: 모든 JS 데이터 타입을 전달할 수 있습니다.

---

### 4. Props 전달 방법 실습

#### (1) 기본 Props 전달 (문자열 및 숫자)
- 문자열은 따옴표(`""`), 숫자는 중괄호(`{}`)를 사용합니다.

```jsx
// ParentComp.jsx
import ChildComp from "./ChildComp";

export default function ParentComp() {
  return (
    <section>
      <ChildComp
        alt="React Logo"   // 문자열 전달
        width={100}         // 숫자 전달
        height={100}
      />
    </section>
  );
}

```
---

#### (2) 객체 형태의 Props 전달
- 중괄호 안에 객체를 넣는 형태(`{{ }}`)로 전달합니다. 
- **바깥 중괄호**: JSX 표현식(JavaScript를 쓰겠다)임을 의미합니다.
- **안쪽 중괄호**: 실제 JavaScript 객체(Object)를 의미합니다.

```jsx
// ParentComp.jsx
import ChildComp from "./ChildComp";
import reactLogo from "../assets/react.svg";

export default function ParentComp() {
  return (
    <ChildComp
      imageInfo={{
        src: reactLogo,
        alt: "React",
      }}
      size={100}
    />
  );
}
```

---

### 5. JSX 내에서 JavaScript 활용
- 중괄호 `{}`를 사용하면 마크업 안에서 동적인 데이터나 함수 호출 결과를 출력할 수 있습니다.
```jsx
// /src/components/UseJsx.jsx
export default function UseJsx() {
  const name = "React";

  function formatDate(date) {
    return new Intl.DateTimeFormat("en-US", { 
      weekday: "long" 
    }).format(date);
  }

  return (
    <>
      <h1>Hello, {name}!</h1>
      <p>Today is {formatDate(new Date())}</p>
    </>
  );
}
```
---

### 6. 실습 단계 및 코드 작성 팁
- **Step 1~2**: 실습에 사용할 `ParentComp`(부모)와 `ChildComp`(자식)를 생성하고, `App.jsx`에서 부모 컴포넌트를 호출하도록 설정합니다.
- **Step 3~6**: `ParentComp`에서 이미지의 `size`나 `alt` 정보를 객체 형태로 전달하고, 자식 컴포넌트에서 이를 받아 렌더링 정보를 수정하는 실습을 진행합니다.
- **가독성 팁**: 속성(Props)이 많을 경우 한 줄로 길게 쓰기보다 **한 줄에 하나씩 분리**하여 작성하는 것이 유지보수와 협업 시 가독성 측면에서 훨씬 유리합니다.

---

### ✅ 5주차 핵심 요약
- **Props**는 부모에서 자식으로 데이터를 전달하는 **유일한 통로**입니다.
- **전달 문법**: 데이터 타입에 따라 구분합니다. (문자열: `" "`, 그 외 숫자/객체/함수: `{ }`)
- **읽기 전용**: 자식은 받은 Props를 읽을 수는 있지만 **절대 직접 변경해서는 안 됩니다.**
- **JSX의 힘**: 중괄호 `{}`는 JavaScript의 강력한 로직을 마크업(HTML)에 결합하는 핵심 도구입니다.

## 📅 4주차: Vite 업데이트와 컴포넌트 설계의 기초 (3월 25일)

### 1. Vite의 트랜스파일러 변경사항 (2026.03.19 이후)
- **주요 변경점**: 2026년 3월 19일 이후 Vite로 프로젝트 생성 시 트랜스파일러 선택 메뉴에서 `JavaScript + SWC` 옵션이 제거되었습니다.
- **기본값 변경**: 이전에는 JavaScript 선택 시 Babel이 설정되었으나, 현재는 **Oxc가 기본 트랜스파일러로 설정**됩니다.
- **프로젝트 생성 방법**:
  - ❌ 이전 방식 (더 이상 사용 불가): `npm create vite@latest my-app --template react-swc`
  - ✅ **현재 방식**: `npm create vite@latest my-app --template react` (인터랙티브 메뉴에서 JavaScript 선택)
- **설정 확인**: `vite.config.js` 파일에서 `plugins: [react()]` 설정을 통해 기본 Oxc 기반 설정을 확인할 수 있습니다.

---

### 2. 차세대 도구 비교: Oxc vs SWC
두 도구 모두 Rust 언어로 작성되어 JavaScript 기반의 Babel보다 압도적으로 빠릅니다.

- **SWC (Speedy Web Compiler)**:
  - 용도: Babel 대체 컴파일러 및 번들러.
  - 특징: 트랜스파일링(TS→JS)과 번들링에 특화되어 있으며 Next.js 등에서 기본 사용 중.
- **Oxc (Oxidation Compiler)**:
  - 용도: ESLint, Prettier, TS 트랜스파일러 등을 모두 통합하려는 고성능 도구 모음.
  - **특징**: 
    - 파싱(Parsing) 속도가 **SWC보다 3배 이상 빠름**.
    - ESLint보다 **100배 이상 빠른** 린팅(Linting) 성능 제공.
- **권장사항**: 특별한 사유가 없다면 속도가 더 빠른 Oxc(기본값)를 사용하는 것이 권장됩니다.

---

### 3. 컴포넌트의 생성 및 사용
- **생성 단계**:
  1. 컴포넌트 이름과 동일한 이름의 파일 생성 (`PascalCase`).
  2. `export default` 키워드를 사용하여 외부에서 쓸 수 있게 함.
  3. `function` 본문 내부(중괄호 `{}`)에 컴포넌트 로직 구현.
  4. `return` 구문 소괄호 `()` 내부에 반환할 JSX 마크업 작성.
- **사용 방법**:
  - 사용하려는 파일 상단에서 컴포넌트를 `import` 함.
  - 임포트한 이름을 `<컴포넌트명 />` 형태로 호출하여 사용.

---

### 4. 컴포넌트 중첩(Nesting)의 개념과 실습
- **개념**: 특정 컴포넌트를 다른 컴포넌트 **안에서 호출**하는 것을 의미합니다.
- **⚠️ 주의 (중요)**: 컴포넌트 **내부에서 다른 컴포넌트를 '선언(정의)'하는 것은 잘못된 방법**입니다. 반드시 최상위 레벨에서 각각 선언하고 호출만 중첩해야 합니다.
- **디렉토리 관리**:
  - 컴포넌트가 많아지면 `/src/components/` 디렉토리를 생성하여 별도로 관리합니다.
  - **예시 구조**: `Gallery` 컴포넌트 내부에 `Profile` 컴포넌트를 중첩하고, 최종적으로 `App.jsx`에서 `Gallery`를 렌더링함.



---

### 5. 컴포넌트 분리 및 관리 권장사항
- **파일 분리**: 프로젝트 루트에 파일이 너무 많아지지 않도록 기능별로 분리합니다.
- **루트 컴포넌트(`App.jsx`)의 역할**: 
  - 최종적으로 렌더링할 컴포넌트들을 모아두는 '그릇' 역할로 사용합니다.
  - 가능하면 `App.jsx` 내부에서 새로운 컴포넌트를 직접 정의하지 않는 것이 좋습니다.
- **계층 구조**: `/src/components/` 내부에 기능별 하위 디렉토리를 만들어 체계적으로 관리합니다.

---

### 6. Export와 Import (가져오기 및 내보내기)
| 구분 | Default Export | Named Export |
| :--- | :--- | :--- |
| **선언 방식** | `export default function Button() {}` | `export function Button() {}` |
| **가져오기** | `import Button from './button.js'` | `import { Button } from './button.js'` |
| **특징** | 파일당 **단 하나**만 가능 | 한 파일에 **여러 개** 가능 |

- **Named Export 활용법**:
  - 여러 개 가져오기: `import { Com1, Com2 } from './files.js'`
  - 이름 변경하기: `import { Button as MyButton } from './button.js'`
  - 전체 가져오기(Namespace): `import * as All from './files.js'`

---

### 7. JSX(JavaScript XML)의 3가지 규칙
React 컴포넌트는 JavaScript 함수이며, 마크업 작성을 위해 JSX 문법을 따릅니다.

1. **하나의 루트 엘리먼트로 반환**: 반드시 하나의 태그로 감싸야 하며, 불필요한 태그를 줄이려면 Fragment(`<>...</>`)를 사용합니다.
2. **모든 태그 닫기**: `<br />`이나 `<img>` 같이 닫는 태그가 없는 HTML 태그도 반드시 끝에 `/`를 넣어 닫아줘야 합니다.
3. **카멜 케이스(camelCase) 속성 사용**: HTML의 `class`는 `className`으로, `onclick`은 `onClick`으로 작성합니다.

- **표기법 정리**:
  - **파스칼 케이스(PascalCase)**: 컴포넌트 이름 (예: `HeaderContent`)
  - **카멜 케이스(camelCase)**: JSX 속성 이름 (예: `tabIndex`, `strokeWidth`)

## 📅 2주차: Node.js 기초 및 React의 핵심 철학 (3월 11일)

### 1. Node.js란 무엇인가?
* **탄생 배경**: 2009년 라이언 달(Ryan Dahl) 개발. 기존 Apache 서버의 **블로킹(Blocking)** 문제를 해결하기 위해 **비동기(Asynchronous)** 방식의 서버 필요성 대두.
* **핵심 엔진**: Chrome의 **V8 엔진** (당시 가장 빠른 브라우저 엔진).

#### 🎯 주요 활용 분야
| 분야 | 설명 | 예시 |
| :--- | :--- | :--- |
| **웹 서버** | REST API, GraphQL API 개발 | Express.js, Nest.js |
| **실시간 앱** | 채팅, 알림, 스트리밍 | Socket.io |
| **서버리스** | FaaS(Function as a Service) | AWS Lambda |
| **CLI 도구** | 커맨드라인 유틸리티 | npm, npx |

#### ✅ 장점과 ❌ 단점
* **장점**: 비동기 논 블로킹 I/O 성능, JS 풀스택 개발 가능, 방대한 npm 생태계.
* **단점**: 싱글 스레드 기반이라 CPU 집약 작업에 부적합, 콜백 지옥(Callback Hell) 위험, npm 패키지 보안 취약점 주의.

---

### 2. React의 진화와 방향
* **역사**: 2013년 5월 29일 Facebook의 Jordan Walke가 공개.
* **정체성 변화**: 단순 "JS 라이브러리"에서 **"웹 및 네이티브 UI를 위한 표준 라이브러리"**로 진화.

---

### 3. 개발 방식의 차이: 명령형 vs 선언형

#### ❌ React 이전 (명령형)
* **방식**: "어떻게(How)" 할지를 명시. 직접 DOM을 선택하고 조작.
* **문제**: 상태 추적이 어렵고 사이드 이펙트(Side Effect) 발생 위험이 높음.

#### ✅ React 방식 (선언형)
* **핵심 공식**: `UI = f(state)` (UI는 상태의 결과물)
* **특징**: 화면을 직접 고치지 않고 **상태(state)를 변경**하면 React가 자동으로 UI를 재계산(렌더링)함.

---

### 4. 렌더링(Rendering)의 진실
> "렌더링은 단순히 화면을 그리는 것이 아니라, **현재 상태를 기준으로 UI를 다시 계산하는 과정**입니다."

1.  **상태 변경** 발생
2.  새로운 화면 모습 **계산**
3.  이전 결과와 **비교 (Diffing)**
4.  필요한 **DOM 변경만 수행** (최적화)

---

### 5. React 전문가의 3가지 조건
1.  **State**: "왜 이 상태가 여기 위치해야 하는가?" 설명 가능
2.  **Rendering**: "왜 컴포넌트가 다시 그려지는가?" 이해
3.  **Effect**: "Effect가 왜 필요한지(혹은 필요 없는지)" 판단 가능

## 📅 1주차: GitHub를 활용한 버전 관리와 협업 기초

### 1. 버전 관리 시스템 (VCS)의 이해
- **개요**: 프로젝트의 변경 사항을 기록하고, 특정 시점으로 되돌리거나 여러 사람과 동시에 작업하기 위해 사용합니다.
- **Git과 GitHub의 차이**:
  - **Git**: 내 컴퓨터(로컬)에서 소스코드 버전을 관리하는 도구입니다.
  - **GitHub**: Git으로 관리하는 프로젝트를 클라우드에 올리고 다른 사람과 공유/협업하는 서비스입니다.

### 2. 핵심 Git 명령어 (CLI)
터미널이나 VS Code 내장 터미널에서 자주 사용하는 기본 명령어입니다.

- `git init`: 새로운 Git 저장소를 생성합니다.
- `git clone [URL]`: 원격 저장소(GitHub)에 있는 프로젝트를 내 컴퓨터로 복사해옵니다.
- `git add .`: 변경된 모든 파일을 스테이징 영역(Staging Area)에 올립니다.
- `git commit -m "메시지"`: 변경 사항을 확정 짓고 기록을 남깁니다.
- `git push`: 로컬의 변경 사항을 원격 저장소(GitHub)로 전송합니다.
- `git pull`: 원격 저장소의 최신 변경 내용을 내 컴퓨터로 가져옵니다.

### 3. GitHub 협업 워크플로우
- **Issue**: 프로젝트에서 해결해야 할 작업, 버그, 건의사항 등을 등록하고 관리합니다.
- **Branch**: 독립적인 작업 공간을 만들어 원본 코드에 영향을 주지 않고 새로운 기능을 개발합니다.
- **Pull Request (PR)**: 내가 작업한 내용을 원본 코드(Main)에 합쳐달라고 요청하는 단계로, 팀원의 코드 리뷰를 거칩니다.
